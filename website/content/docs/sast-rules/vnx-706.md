---
title: "VNX-706 – Use of Incorrectly-Resolved Name or Reference (Variable Shadowing)"
description: "Detect security-critical variable names like 'password' or 'token' re-declared in local scope, potentially shadowing outer-scope credentials used in security checks."
---

## Overview

This rule flags re-declarations of security-sensitive variable names (password, token, secret, api_key) in inner scopes where they may shadow an outer-scope credential. When a local variable shadows the intended credential, security checks may operate on the wrong value — for example, checking an empty local password instead of the submitted one. This maps to [CWE-706: Use of Incorrectly-Resolved Name or Reference](https://cwe.mitre.org/data/definitions/706.html).

**Severity:** Low | **CWE:** [CWE-706](https://cwe.mitre.org/data/definitions/706.html)

## Why This Matters

Variable shadowing in security-critical code can cause authentication checks to pass trivially. If a `password` variable is declared inside a conditional block with an empty string value, any subsequent comparison against it will succeed with any input.

## What Gets Flagged

```python
# FLAGGED: local 'password' shadows outer scope
def login(username, password):
    user = db.get_user(username)
    if user:
        password = ""  # accidentally shadows the parameter!
        if password == user.password_hash:  # always fails or trivially passes
            return True
```

```go
// FLAGGED: short declaration shadows outer password
func verify(password string) bool {
    if condition {
        password := getDefault()  // shadows the parameter
        return check(password)    // wrong password checked
    }
    return check(password)
}
```

## Remediation

- Use distinct names for inner-scope variables.
- In Go, use `=` instead of `:=` when assigning to an existing variable.
- In Python, avoid reassigning parameter names inside function bodies.
