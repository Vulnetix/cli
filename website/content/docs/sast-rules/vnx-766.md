---
title: "VNX-766 – Critical Data Element Declared Public"
description: "Detect Java fields containing passwords, API keys, or tokens declared as public, allowing direct access bypassing encapsulation."
---

## Overview

This rule flags Java fields with security-sensitive names (password, apiKey, token, secret, privateKey) declared as `public`. Public fields allow any class in the JVM to read or modify the value directly, bypassing encapsulation, validation, and audit logging. This maps to [CWE-766: Critical Data Element Declared Public](https://cwe.mitre.org/data/definitions/766.html).

**Severity:** Medium | **CWE:** [CWE-766](https://cwe.mitre.org/data/definitions/766.html)

## Why This Matters

Public credential fields can be read via reflection by any code in the JVM, including third-party libraries. They also appear in serialized object dumps, log output of object.toString() methods, and are accessible to any subclass or calling code without audit trail.

## What Gets Flagged

```java
// FLAGGED: public password field
public class UserService {
    public String password = "";
    public String apiKey = "";
    public static String SECRET_TOKEN = "...";
}
```

## Remediation

```java
// SAFE: private fields with controlled access
public class UserService {
    private String password;
    private final String apiKey;

    // Optional: use char[] for passwords so they can be zeroed
    private char[] password;

    public void setPassword(char[] newPassword) {
        // validate before storing
        this.password = Arrays.copyOf(newPassword, newPassword.length);
    }

    public boolean verifyPassword(char[] submitted) {
        return Arrays.equals(this.password, submitted);
    }

    // Never provide a getPassword() that returns the actual value
}
```

Use a secrets management solution (AWS Secrets Manager, HashiCorp Vault) rather than storing credentials in fields at all.
