---
title: "VNX-580 – clone() Method Without super.clone()"
description: "Detect Java clone() overrides that do not call super.clone(), violating the clone contract."
---

## Overview

Java's `clone()` contract requires that `super.clone()` be called to use the native shallow-copy mechanism. Implementing `clone()` without calling `super.clone()` creates objects incorrectly, potentially sharing state between the original and clone in unexpected ways.

**Severity:** Low | **CWE:** [CWE-580 – clone() Method Without super.clone()](https://cwe.mitre.org/data/definitions/580.html)

## Why This Matters

Improperly cloned security objects (credentials, permission sets, sessions) may share mutable state with their originals. Modifying the clone may unintentionally modify the original, or vice versa, creating privilege escalation or data leakage bugs.

## What Gets Flagged

```java
// FLAGGED: clone() without super.clone()
@Override
public Object clone() {
    MyObject obj = new MyObject();  // Wrong: should call super.clone()
    obj.data = this.data;
    return obj;
}
```

## Remediation

Always call `super.clone()` first:

```java
// SAFE: Correct clone implementation
@Override
public Object clone() throws CloneNotSupportedException {
    MyObject obj = (MyObject) super.clone();
    // Deep copy mutable fields if needed
    obj.list = new ArrayList<>(this.list);
    return obj;
}
```
