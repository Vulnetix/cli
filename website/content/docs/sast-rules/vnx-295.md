## Overview

This rule flags code that disables or bypasses TLS/SSL certificate validation, removing the only guarantee that a connection reaches the intended server. Patterns include Python's `verify=False`, Go's `InsecureSkipVerify: true`, Node.js's `rejectUnauthorized: false`, PHP's `CURLOPT_SSL_VERIFYPEER => false`, Ruby's `VERIFY_NONE`, and Java TrustManagers that accept all certificates. This maps to [CWE-295: Improper Certificate Validation](https://cwe.mitre.org/data/definitions/295.html).

**Severity:** Critical | **CWE:** [CWE-295 – Improper Certificate Validation](https://cwe.mitre.org/data/definitions/295.html)

## Why This Matters

TLS certificate validation serves two purposes: it proves the server possesses the private key for the certificate, and it proves the certificate was issued by a trusted CA for the correct hostname. Disabling verification eliminates both guarantees. On any network path between client and server (corporate proxy, coffee shop Wi-Fi, compromised ISP, BGP hijack), an attacker can intercept the TLS connection, present a self-signed certificate, and read or modify all traffic in both directions. This is a classic man-in-the-middle (MITM) attack. When authentication credentials, session tokens, or sensitive data flow over the connection, certificate bypass turns a network-adjacent attacker into a full account compromise. Developers often disable verification to work around certificate errors in development or testing — and then accidentally commit the bypass to production code.

## What Gets Flagged

- **Python**: `verify=False` (requests), `ssl._create_unverified_context(`, `ssl.CERT_NONE`, `check_hostname=False`
- **Go**: `InsecureSkipVerify: true`
- **Node.js**: `rejectUnauthorized: false`, `NODE_TLS_REJECT_UNAUTHORIZED`
- **PHP**: `CURLOPT_SSL_VERIFYPEER, false`, `CURLOPT_SSL_VERIFYPEER, 0`, `CURLOPT_SSL_VERIFYHOST, 0`
- **Ruby**: `VERIFY_NONE`, `verify_mode: OpenSSL::SSL::VERIFY_NONE`
- **Java**: `X509TrustManager` with empty/null `checkServerTrusted`

```python
# FLAGGED: certificate verification disabled in Python
import requests
response = requests.get("https://internal.example.com/api", verify=False)

import ssl
ctx = ssl._create_unverified_context()
```

```go
// FLAGGED: InsecureSkipVerify in Go
tr := &http.Transport{
    TLSClientConfig: &tls.Config{InsecureSkipVerify: true},
}
```

```javascript
// FLAGGED: rejectUnauthorized disabled in Node.js
const https = require('https');
const options = {
    hostname: 'example.com',
    rejectUnauthorized: false,
};
```

```php
// FLAGGED: CURLOPT_SSL_VERIFYPEER disabled in PHP
curl_setopt($ch, CURLOPT_SSL_VERIFYPEER, false);
curl_setopt($ch, CURLOPT_SSL_VERIFYHOST, 0);
```

```ruby
# FLAGGED: VERIFY_NONE in Ruby
http.verify_mode = OpenSSL::SSL::VERIFY_NONE
```

## Remediation

1. **Never disable certificate verification in production code.** If you encounter a certificate error, fix the certificate — do not disable verification.

```go
// SAFE: proper TLS config in Go (default behavior, no InsecureSkipVerify)
client := &http.Client{}
// Default transport validates certificates automatically
```

```python
# SAFE: specify custom CA if using a private CA
import requests
response = requests.get(url, verify='/etc/ssl/certs/my-ca.crt')
```

2. **For development with self-signed certs**, trust the specific certificate in your local trust store, or use a tool like `mkcert` to generate locally-trusted certificates.

3. **For internal services**, provision certificates from your organization's internal CA and add that CA to the trust store rather than disabling verification.

4. **Audit infrastructure.** Use `grep -r InsecureSkipVerify` and equivalent searches in CI pipelines to prevent certificate bypass from reaching production.
