---
title: "VNX-762 – Mismatched Memory Management Routines"
description: "Detect C++ code mixing malloc/free with new/delete, or using delete instead of delete[] for array allocations."
---

## Overview

This rule flags C++ code that mixes memory management routines: `malloc` with `delete`, `new` with `free`, or `new[]` with plain `delete` (should be `delete[]`). These mismatches cause undefined behaviour — typically heap corruption — because the allocator's bookkeeping for C malloc and C++ new differs. This maps to [CWE-762: Mismatched Memory Management Routines](https://cwe.mitre.org/data/definitions/762.html).

**Severity:** High | **CWE:** [CWE-762](https://cwe.mitre.org/data/definitions/762.html)

## Why This Matters

C++ allocators store metadata (size, type information for destructors) differently than C malloc. Freeing a `new`'d object with `free()` skips destructor calls and corrupts the heap. Freeing a `new[]` array with `delete` (not `delete[]`) has undefined behaviour and may corrupt the heap or skip destructors for all but the first element.

## What Gets Flagged

```cpp
// FLAGGED: new[] then plain delete (should be delete[])
int *arr = new int[100];
// ...
delete arr;   // WRONG: should be delete[] arr

// FLAGGED: malloc then delete (should be free)
char *buf = (char*) malloc(256);
delete buf;   // WRONG: should be free(buf)

// FLAGGED: free() in a file that uses new[] allocations
int *data = new int[n];
free(data);   // WRONG: should be delete[] data
```

## Remediation

```cpp
// SAFE: match allocation and deallocation routines
int *arr = new int[100];
delete[] arr;  // matches new[]

char *buf = (char*) malloc(256);
free(buf);     // matches malloc

// BEST: prefer RAII and smart pointers to avoid manual memory management
#include <memory>
auto arr = std::make_unique<int[]>(100);  // automatically freed
std::vector<int> vec(100);               // no manual management needed
```
