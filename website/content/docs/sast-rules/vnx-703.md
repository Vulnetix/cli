title: "VNX-703 – Improper Check or Handling of Exceptional Conditions"
description: "Detect broad catch-all exception handlers that swallow errors without logging, hiding security events and making the system impossible to debug."
---

## Overview

This rule flags exception handlers that catch all exceptions with a broad catch-all clause and do nothing meaningful with them — swallowing errors silently with `pass`, `nil`, or an empty body. When exceptions are suppressed, security-relevant events go undetected, authentication failures are hidden from monitoring, and bugs become impossible to diagnose. This maps to [CWE-703: Improper Check or Handling of Exceptional Conditions](https://cwe.mitre.org/data/definitions/703.html).

**Severity:** Medium | **CWE:** [CWE-703](https://cwe.mitre.org/data/definitions/703.html)

## Why This Matters

Silent error swallowing is one of the most dangerous anti-patterns in security-sensitive code. When authentication throws an exception that is silently caught and ignored, a login function may continue as if authentication succeeded. When a cryptographic operation fails silently, the program may proceed with an empty or default key. Exception suppression is also a major obstacle to incident response: if errors aren't logged, investigators have no evidence to work with.

## What Gets Flagged

```python
# FLAGGED: bare except with pass
try:
    authenticate_user(username, password)
except Exception:
    pass  # Authentication error silently ignored

# FLAGGED: bare except clause
try:
    validate_token(token)
except:
    pass
```

```java
// FLAGGED: catch-all exception handler
try {
    processPayment(amount);
} catch (Exception e) {
    // silently ignored
}
```

```javascript
// FLAGGED: empty catch block
try {
    verifySignature(data, sig);
} catch (e) {}
```

## Remediation

1. **Log and rethrow or handle specifically.** At minimum, log the exception with context before deciding what to do with it.

```python
# SAFE: log the exception and handle appropriately
try:
    authenticate_user(username, password)
except AuthenticationError as e:
    logger.warning("Authentication failed for user %s: %s", username, str(e))
    raise
except Exception as e:
    logger.error("Unexpected error during authentication: %s", str(e), exc_info=True)
    raise
```

```java
// SAFE: log and handle or rethrow
try {
    processPayment(amount);
} catch (PaymentException e) {
    logger.error("Payment processing failed: {}", e.getMessage(), e);
    throw new ServiceException("Payment failed", e);
}
```

2. **Catch specific exception types.** Avoid catching `Exception`, `Throwable`, or bare `except` — catch only the exceptions you can actually handle.

3. **Never use exceptions for flow control** in a way that suppresses security errors.
