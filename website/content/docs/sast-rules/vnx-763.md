title: "VNX-763 – Release of Invalid Pointer or Reference"
description: "Detect C free() calls on string literals, stack addresses, or other non-heap memory, and double-free patterns."
---

## Overview

This rule flags `free()` calls on memory that was not heap-allocated — string literals, stack addresses (`&localvar`), or static memory — and double-free patterns where the same pointer is freed twice consecutively. Freeing invalid pointers corrupts the heap allocator, typically causing crashes or exploitable memory corruption. This maps to [CWE-763: Release of Invalid Pointer or Reference](https://cwe.mitre.org/data/definitions/763.html).

**Severity:** High | **CWE:** [CWE-763](https://cwe.mitre.org/data/definitions/763.html)

## Why This Matters

Double-free vulnerabilities are a well-studied exploit class. When a pointer is freed twice, the allocator may link the freed chunk into the free list twice, allowing an attacker who can control allocation patterns to overwrite arbitrary memory. CVE-2019-11043 (PHP remote code execution) and many others have exploited double-free conditions.

## What Gets Flagged

```c
// FLAGGED: free of string literal
char *name = "admin";
free(name);  // WRONG: string literals are in read-only memory

// FLAGGED: free of stack address
void func() {
    int x = 42;
    int *ptr = &x;
    free(ptr);  // WRONG: ptr points to stack memory
}

// FLAGGED: double free
char *buf = malloc(256);
process(buf);
free(buf);
free(buf);  // WRONG: already freed
```

## Remediation

```c
// SAFE: set pointer to NULL after free to prevent double-free
char *buf = malloc(256);
process(buf);
free(buf);
buf = NULL;  // subsequent free(NULL) is a no-op

// SAFE: only free heap-allocated memory
const char *literal = "admin";    // do NOT free this
char *copy = strdup("admin");     // safe to free - strdup uses malloc
free(copy);
copy = NULL;
```
