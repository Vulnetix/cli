---
title: "VNX-JAVA-016 – Java weak PRNG (java.util.Random) used for security-sensitive value"
description: "Detects use of java.util.Random and Math.random() for generating tokens, session IDs, nonces, passwords, or keys — values that must be unpredictable but are not when produced by a non-cryptographic PRNG."
---

## Overview

This rule flags any use of `java.util.Random` (via `new Random()` or `new java.util.Random()`) and `Math.random()` in Java source files. Both are pseudo-random number generators (PRNGs) whose output is fully determined by a 48-bit seed, making them computationally predictable. The JDK's implementation seeds `java.util.Random` from the current time and a counter, meaning an attacker who can guess the approximate seed time can reproduce every value the generator has produced or will produce.

When predictable random values are used for security-critical purposes — session tokens, password reset codes, CSRF nonces, cryptographic IVs, or API keys — an attacker who can observe a few outputs can reconstruct the internal state and predict past and future values. This breaks the fundamental security guarantee that these values rely on: secrecy through unpredictability.

`java.security.SecureRandom` is the correct alternative. It is backed by the operating system's entropy source (e.g., `/dev/urandom` on Linux, `CryptGenRandom` on Windows), which provides genuine unpredictability that no amount of computation can feasibly reverse. This vulnerability corresponds to CWE-330 (Use of Insufficiently Random Values).

**Severity:** High | **CWE:** [CWE-330 – Use of Insufficiently Random Values](https://cwe.mitre.org/data/definitions/330.html)

## Why This Matters

The predictability of `java.util.Random` has been exploited in real-world attacks against web applications. A well-known class of attack targets session token generation: if a server generates tokens by calling `new Random().nextLong()`, an attacker who can register an account and receive a few tokens can run a lattice-reduction attack to recover the 48-bit internal state, then enumerate all possible tokens for every other active session. This has been demonstrated against several Java application servers.

Password reset flows are equally at risk. If a "forgot password" link embeds a token generated by `Math.random()`, an attacker can request thousands of resets in a short window, collect the issued tokens, reconstruct the PRNG state, and then predict the token issued for a victim's account — executing a full account takeover without ever intercepting network traffic.

Beyond session hijacking, weak random values used for IVs or nonces in cryptographic modes (CBC, CTR, GCM) degrade encryption to a state where chosen-plaintext attacks become practical. What appears to be a minor API choice has cascading consequences across the entire security posture of the application.

## What Gets Flagged

```java
// FLAGGED: java.util.Random used to generate a session token
import java.util.Random;

public class SessionManager {
    private static final Random RNG = new Random();

    public String generateSessionToken() {
        return Long.toHexString(RNG.nextLong()); // predictable!
    }
}

// FLAGGED: Math.random() used for a password reset code
public String generateResetCode() {
    int code = (int) (Math.random() * 999999); // predictable!
    return String.format("%06d", code);
}
```

## Remediation

1. Replace every instance of `new Random()` and `Math.random()` in security-sensitive paths with `java.security.SecureRandom`.
2. For token generation, call `nextBytes()` on a `SecureRandom` instance and hex-encode or Base64-encode the result rather than using numeric output.
3. Treat `SecureRandom` as a shared, long-lived instance — it is thread-safe and expensive to seed. Avoid creating a new instance per request.
4. For generating UUIDs suitable for use as nonces or correlation IDs, use `UUID.randomUUID()`, which internally uses `SecureRandom`.

```java
// SAFE: SecureRandom used for session token generation
import java.security.SecureRandom;
import java.util.Base64;

public class SessionManager {
    private static final SecureRandom SECURE_RNG = new SecureRandom();

    public String generateSessionToken() {
        byte[] token = new byte[32]; // 256 bits
        SECURE_RNG.nextBytes(token);
        return Base64.getUrlEncoder().withoutPadding().encodeToString(token);
    }

    public String generateResetCode() {
        byte[] buf = new byte[4];
        SECURE_RNG.nextBytes(buf);
        // Use modulo on a 32-bit positive integer for a 6-digit code
        int code = (int) (Integer.toUnsignedLong(
                buf[0] << 24 | (buf[1] & 0xFF) << 16 | (buf[2] & 0xFF) << 8 | (buf[3] & 0xFF))
                % 1_000_000L);
        return String.format("%06d", code);
    }
}
```
