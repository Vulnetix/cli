---
title: "VNX-1037 – Processor Optimization Removal of Security-critical Code"
description: "Detect memset() calls on sensitive variables that may be eliminated by compiler dead-store optimization."
---

## Overview

When a compiler's dead-store elimination optimization determines that a memory write result is never subsequently read, it may remove the write. The classic case is `memset(password_buf, 0, len); free(password_buf);` — the compiler sees that `password_buf` is freed immediately after, so the zeroing has no observable effect and can be removed. The sensitive data remains in the freed heap block.

**Severity:** Medium | **CWE:** [CWE-1037 – Processor Optimization Removal or Modification of Security-critical Code](https://cwe.mitre.org/data/definitions/1037.html)

## Why This Matters

After `free()`, the memory containing sensitive data (passwords, keys, tokens) is returned to the allocator and may be read by another allocation or a heap dump. An attacker with memory-read access (via another vulnerability) can extract sensitive values that should have been zeroed.

## What Gets Flagged

```c
// FLAGGED: memset may be optimized away
memset(password, 0, password_len);
free(password);

// FLAGGED: memset on sensitive var without secure function
char secret[32];
memset(secret, 0, 32);  // before return, compiler may remove
```

## Remediation

Use compiler-safe zeroing functions:

```c
// SAFE: explicit_bzero (POSIX, guaranteed not optimized away)
explicit_bzero(password, password_len);
free(password);

// SAFE: Windows
SecureZeroMemory(password, password_len);

// SAFE: C11
memset_s(password, sizeof(password), 0, sizeof(password));
```
