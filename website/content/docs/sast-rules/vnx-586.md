---
title: "VNX-586 – Explicit Call to finalize()"
description: "Detect explicit calls to Java's finalize() method and finalize() overrides that don't call super.finalize()."
---

## Overview

Java's `finalize()` method is intended to be called only by the garbage collector, not by application code. Explicit calls to `obj.finalize()` can trigger finalization before the object is garbage-collected, potentially leading to double-finalization, resource corruption, and security issues.

**Severity:** Low | **CWE:** [CWE-586 – Explicit Call to Finalize()](https://cwe.mitre.org/data/definitions/586.html)

## Why This Matters

Double finalization of objects that hold cryptographic keys, database connections, or file handles can corrupt the associated resources. If `finalize()` releases sensitive memory back to the JVM heap prematurely, the data may be observable to other code.

## What Gets Flagged

```java
// FLAGGED: Explicit finalize() call
obj.finalize();

// FLAGGED: finalize() override without super.finalize()
@Override
protected void finalize() {
    releaseResources();
    // Missing: super.finalize()
}
```

## Remediation

Use `AutoCloseable` and try-with-resources instead:

```java
// SAFE: Implement AutoCloseable for deterministic cleanup
public class SecureResource implements AutoCloseable {
    @Override
    public void close() {
        releaseResources();
    }
}

// Usage
try (SecureResource r = new SecureResource()) {
    r.use();
}  // close() called deterministically
```
