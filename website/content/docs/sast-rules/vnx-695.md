## Overview

Low-level functionality like Go's `unsafe.Pointer` and C#'s `unsafe` blocks bypass the type system and memory safety guarantees provided by the runtime. While sometimes necessary for FFI or performance-critical code, these constructs can introduce memory corruption, type confusion, and security vulnerabilities that are impossible to catch with normal static analysis.

**Severity:** Medium | **CWE:** [CWE-695 – Use of Low-Level Functionality](https://cwe.mitre.org/data/definitions/695.html)

## Why This Matters

Go's `unsafe.Pointer` allows arbitrary pointer arithmetic that can corrupt adjacent memory, bypass bounds checking, and create use-after-free conditions. C#'s `unsafe` keyword disables the CLR's memory safety guarantees for the enclosed block. Both patterns have been used to bypass security boundaries in sandboxed environments.

## What Gets Flagged

```go
// FLAGGED: unsafe.Pointer usage — bypasses Go type safety
import "unsafe"

func manipulate(p *int) {
    ptr := unsafe.Pointer(p)
    // Can now cast to arbitrary type or do pointer arithmetic
    offsetPtr := unsafe.Pointer(uintptr(ptr) + 8)
}
```

```csharp
// FLAGGED: unsafe block in C# — bypasses CLR memory safety
unsafe {
    int* ptr = &value;
    *(ptr + 1) = 0xDEADBEEF;  // arbitrary memory write
}
```

## Remediation

Avoid `unsafe` operations unless absolutely necessary, and isolate them if required:

```go
// SAFE: Use reflect for introspection instead of unsafe
import "reflect"
v := reflect.ValueOf(obj)
field := v.FieldByName("SecretField")

// If unsafe is genuinely required (e.g., CGo FFI):
// - Isolate in a separate package
// - Add security review comment explaining why it's needed
// - Fuzz test the unsafe code thoroughly
```

```csharp
// SAFE: Use managed code; only use unsafe for specific P/Invoke scenarios
// And only with proper bounds checking:
unsafe {
    if (index < 0 || index >= length) throw new IndexOutOfRangeException();
    return *(ptr + index);
}
```
