---
title: "VNX-705 – Incorrect Control Flow Scoping"
description: "Detect return statements inside finally blocks that suppress exceptions and override prior return values, causing incorrect control flow."
---

## Overview

This rule flags `return` statements inside `finally` blocks. A return in a finally block suppresses any exception thrown in the try/catch body and overrides any prior return value, making it impossible for callers to detect that an error occurred. This maps to [CWE-705: Incorrect Control Flow Scoping](https://cwe.mitre.org/data/definitions/705.html).

**Severity:** Low | **CWE:** [CWE-705](https://cwe.mitre.org/data/definitions/705.html)

## Why This Matters

When a security-sensitive operation throws an exception and the finally block returns a success value, the caller receives a success response from a failed operation. This can cause authentication functions to appear to succeed even when they fail.

## What Gets Flagged

```java
// FLAGGED: return inside finally suppresses exception
public boolean authenticate(String user, String pass) {
    try {
        return verifyCredentials(user, pass);
    } catch (AuthException e) {
        throw new SecurityException("Auth failed", e);
    } finally {
        return true;  // ALWAYS returns true, even on exception!
    }
}
```

```python
# FLAGGED: return in finally
def validate():
    try:
        return check_token()
    finally:
        return True  # overrides exception
```

## Remediation

```java
// SAFE: only use finally for cleanup, not for returning values
public boolean authenticate(String user, String pass) {
    try {
        return verifyCredentials(user, pass);
    } catch (AuthException e) {
        throw new SecurityException("Auth failed", e);
    } finally {
        cleanup();  // cleanup only, no return
    }
}
```
