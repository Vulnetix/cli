---
title: "VNX-609 – Double-Checked Locking"
description: "Detect Java double-checked locking implementations missing the volatile keyword on the singleton field."
---

## Overview

The classic Java double-checked locking pattern for lazy singleton initialization is broken without the `volatile` keyword on the instance field. The Java Memory Model permits instruction reordering that can cause a partially-constructed object to be published to other threads.

**Severity:** Medium | **CWE:** [CWE-609 – Double-Checked Locking](https://cwe.mitre.org/data/definitions/609.html)

## Why This Matters

A partially-constructed singleton that holds security configuration (crypto keys, permission lists, LDAP connections) may appear non-null to another thread before its fields are fully initialized, causing security checks to run against corrupt data.

## What Gets Flagged

```java
// FLAGGED: Missing volatile on instance field
private static SecurityManager instance;  // not volatile!

public static SecurityManager getInstance() {
    if (instance == null) {
        synchronized (SecurityManager.class) {
            if (instance == null) {
                instance = new SecurityManager();
            }
        }
    }
    return instance;
}
```

## Remediation

Add the `volatile` keyword:

```java
// SAFE: volatile ensures visibility across threads
private static volatile SecurityManager instance;
```

Or use the initialization-on-demand holder pattern:

```java
// SAFE: Thread-safe without volatile/synchronized
private static class Holder {
    static final SecurityManager INSTANCE = new SecurityManager();
}
public static SecurityManager getInstance() { return Holder.INSTANCE; }
```
