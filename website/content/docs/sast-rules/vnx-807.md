---
title: "VNX-807 – Reliance on Untrusted Inputs in a Security Decision"
description: "Detect authentication or authorization decisions based on user-controlled HTTP headers, query parameters, or request body fields."
---

## Overview

This rule flags security decisions (authentication, authorization, privilege escalation) that rely on values the attacker controls: custom HTTP headers (`X-Admin`, `X-Role`), query parameters (`?bypass_auth=true`, `?is_admin=1`), or request body fields. An attacker can set any header or parameter value. This maps to [CWE-807: Reliance on Untrusted Inputs in a Security Decision](https://cwe.mitre.org/data/definitions/807.html).

**Severity:** High | **CWE:** [CWE-807](https://cwe.mitre.org/data/definitions/807.html)

## Why This Matters

HTTP headers and request parameters are fully attacker-controlled. Using them directly in access control checks means any authenticated (or unauthenticated) user can elevate their privileges by setting a header like `X-Admin: true` or appending `?is_admin=1` to a request. This is a common bug in proxy-based architectures where an internal service trusts headers set by a gateway that an attacker can bypass.

## What Gets Flagged

```javascript
// FLAGGED: trusting X-Admin header for admin check
if (req.headers['x-admin'] === 'true') {
    grantAdminAccess(req);
}
if (req.headers['x-role'] === 'admin') {
    next();
}
```

```python
# FLAGGED: bypass_auth via query parameter
if request.GET.get('bypass_auth') == 'true':
    return redirect(dashboard)

# FLAGGED: is_admin from query string
if request.GET.get('is_admin') == '1':
    grant_admin(user)
```

```php
// FLAGGED: is_admin from POST or GET
if ($_GET['is_admin'] == '1') {
    showAdminPanel();
}
if ($_POST['role'] == 'admin') {
    setAdminPrivileges();
}
```

```java
// FLAGGED: security decision from request header
String role = request.getHeader("X-User-Role");
if ("admin".equals(role)) {
    grantAccess(request);
}
```

## Remediation

```javascript
// SAFE: derive role from authenticated session, not request headers
app.get('/admin', requireAuth, (req, res) => {
    const user = req.session.user;  // set server-side at login
    if (user.role !== 'admin') {
        return res.status(403).json({ error: 'Forbidden' });
    }
    // ... admin logic
});
```

```python
# SAFE: use server-side session for authorization
from functools import wraps

def require_admin(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        user_id = session.get('user_id')
        if not user_id:
            return redirect(url_for('login'))
        user = User.query.get(user_id)
        if not user or user.role != 'admin':
            abort(403)
        return f(*args, **kwargs)
    return decorated
```

```java
// SAFE: check role from authenticated principal, not headers
String role = (String) request.getSession().getAttribute("role");
if (!"admin".equals(role)) {
    response.sendError(HttpServletResponse.SC_FORBIDDEN);
    return;
}
```

Authorization decisions must be based on cryptographically verified session data or signed tokens (JWT with server-side validation), never on unverified request inputs.
