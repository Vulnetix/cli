title: "VNX-613 – Insufficient Session Expiration"
description: "Detect overly long or missing session and JWT token expiry configurations that allow indefinite session reuse."
---

## Overview

Session tokens with excessively long lifetimes (or no expiry at all) remain valid for an attacker who obtains them through XSS, network sniffing, log exfiltration, or a compromised device. CWE-613 covers all cases where a session is not properly invalidated after a reasonable inactivity period or absolute lifetime.

**Severity:** Medium | **CWE:** [CWE-613 – Insufficient Session Expiration](https://cwe.mitre.org/data/definitions/613.html)

## Why This Matters

A stolen session token is as good as a stolen password until the session expires. A one-year Django `SESSION_COOKIE_AGE`, a Java `setMaxInactiveInterval(-1)`, or a JWT issued without an `exp` claim creates a permanent backdoor for anyone who captures the token.

## What Gets Flagged

```python
# FLAGGED: Django session lasting 1 year
SESSION_COOKIE_AGE = 86400 * 365

# FLAGGED: JWT signed without expiry
token = jwt.encode({"user_id": uid}, SECRET)  # no expiresIn
```

```java
// FLAGGED: Session that never times out
session.setMaxInactiveInterval(-1);
```

## Remediation

Set short, explicit expiry values on all session and token types:

```python
# SAFE: 1 hour session
SESSION_COOKIE_AGE = 3600

# SAFE: JWT with 15-minute expiry
token = jwt.sign({"user_id": uid}, SECRET, expiresIn="15m")
```

```java
// SAFE: 30-minute inactivity timeout
session.setMaxInactiveInterval(1800);
```
