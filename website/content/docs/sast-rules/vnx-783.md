description: "Detect mixed && and || operators in security conditions without explicit parentheses, where precedence may cause unexpected evaluation order."
---

## Overview

This rule flags security-sensitive conditional expressions that mix `&&` and `||` operators without explicit parentheses around subexpressions. Because `&&` binds more tightly than `||`, the expression `a == b || c == d && e == f` evaluates as `a == b || (c == d && e == f)` — which may not be the programmer's intent. In authentication and authorization checks, this can cause conditions to always pass. This maps to [CWE-783: Operator Precedence Logic Error](https://cwe.mitre.org/data/definitions/783.html).

**Severity:** Medium | **CWE:** [CWE-783](https://cwe.mitre.org/data/definitions/783.html)

## Why This Matters

A subtly wrong boolean expression in an authentication check can allow any user (or no user) to pass. The classic mistake is `if (isAdmin || hasRole && isAuthenticated)` — which passes if `isAdmin` is true regardless of authentication status.

## What Gets Flagged

```c
// FLAGGED: mixed && and || in admin check
if (isAdmin || hasRole && isAuthenticated) {
    // evaluates as: isAdmin || (hasRole && isAuthenticated)
    // unauthenticated admins pass!
}
```

```java
// FLAGGED: precedence issue in permission check
if (isAuthorized || bypass && isAuthenticated) {
    grantAccess();
}
```

```python
# FLAGGED: mixed operators in admin check
if is_admin or has_permission and is_authenticated:
    # Python: 'and' binds tighter than 'or'
    # Same precedence issue
    pass
```

## Remediation

```java
// SAFE: explicit parentheses make intent unambiguous
if ((isAdmin || hasRole) && isAuthenticated) {
    // user must be authenticated AND (admin or has role)
}

// SAFE: separate conditions for clarity
boolean hasPrivilege = isAdmin || hasRole;
if (hasPrivilege && isAuthenticated) {
    grantAccess();
}
```

```python
# SAFE: explicit parentheses
if (is_admin or has_permission) and is_authenticated:
    pass
```
