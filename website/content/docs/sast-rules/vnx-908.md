---
title: "VNX-908 – Use of Uninitialized Resource"
description: "Detect usage of file handles, sockets, or database connections before successful initialization."
---

## Overview

Using a resource (file handle, socket, database connection, pointer) before it has been successfully opened or initialized produces undefined behaviour. The most common manifestation is a null-pointer dereference when `fopen()` returns NULL due to a permissions error and the code proceeds to call `fread()` without checking.

**Severity:** Medium | **CWE:** [CWE-908 – Use of Uninitialized Resource](https://cwe.mitre.org/data/definitions/908.html)

## Why This Matters

Uninitialized resource use causes crashes that can be exploited for denial-of-service, and in some cases the uninitialized value contains stale data from a prior allocation that leaks sensitive information.

## What Gets Flagged

```c
// FLAGGED: FILE* used without NULL check
FILE *fp;
fopen("config.txt", "r");  // return value ignored
fread(buf, 1, 100, fp);    // fp may be NULL
```

```go
// FLAGGED: No error check
f := os.Open("config.txt")  // error ignored
```

## Remediation

Always check resource initialization before use:

```c
// SAFE
FILE *fp = fopen("config.txt", "r");
if (fp == NULL) {
    perror("fopen"); return -1;
}
fread(buf, 1, 100, fp);
fclose(fp);
```

```go
// SAFE
f, err := os.Open("config.txt")
if err != nil { return err }
defer f.Close()
```
