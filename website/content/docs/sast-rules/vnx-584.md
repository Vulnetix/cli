---
title: "VNX-584 – Return in Finally Block"
description: "Detect return statements inside finally blocks that suppress exceptions from try/catch blocks."
---

## Overview

A `return` statement inside a `finally` block swallows any exception propagating from the `try` or `catch` blocks. Security-critical exceptions (authentication failures, authorization denials, validation errors) are silently discarded, allowing execution to continue as if nothing went wrong.

**Severity:** Medium | **CWE:** [CWE-584 – Return in Finally Block](https://cwe.mitre.org/data/definitions/584.html)

## Why This Matters

If an authentication check throws `AuthException`, a `return` in `finally` suppresses it. The caller receives a normal return value and proceeds with the assumption that authentication succeeded. This is a subtle fail-open vulnerability.

## What Gets Flagged

```java
// FLAGGED: return in finally silences AuthException
try {
    authenticateUser(token);
} catch (AuthException e) {
    throw e;  // This is never actually thrown
} finally {
    return session;  // Swallows the exception!
}
```

```python
# FLAGGED: return in finally
try:
    verify_token(token)
finally:
    return True  # Suppresses any exception from verify_token
```

## Remediation

Never use return in finally blocks:

```java
// SAFE: Let exceptions propagate
Session session = null;
try {
    session = authenticateUser(token);
} catch (AuthException e) {
    logger.error("Auth failed", e);
    throw e;
} finally {
    cleanup();  // Only cleanup, no return
}
return session;
```
