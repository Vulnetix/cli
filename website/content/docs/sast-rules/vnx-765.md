## Overview

This rule flags consecutive unlock calls without an intervening lock. Unlocking an already-unlocked mutex causes a fatal panic in Go (`sync: unlock of unlocked mutex`), a `RuntimeError` in Python, and `IllegalMonitorStateException` in Java. This maps to [CWE-765: Multiple Unlocks of a Lock](https://cwe.mitre.org/data/definitions/765.html).

**Severity:** Medium | **CWE:** [CWE-765](https://cwe.mitre.org/data/definitions/765.html)

## Why This Matters

Panics and runtime errors in security-critical code paths crash the service, enabling denial of service. Double-unlock can occur when error handling paths trigger an Unlock independently of the normal path.

## What Gets Flagged

```go
// FLAGGED: double Unlock()
mu.Lock()
defer mu.Unlock()
// ...
mu.Unlock()  // already deferred, will unlock twice
```

```python
# FLAGGED: consecutive release()
lock.acquire()
lock.release()
lock.release()  # RuntimeError: release unlocked lock
```

## Remediation

```go
// SAFE: use only defer for unlock, not manual + defer
mu.Lock()
defer mu.Unlock()
// never call mu.Unlock() again in the same function
```

```python
# SAFE: use context manager
with lock:
    # critical section
    pass
# lock is automatically released, cannot be double-released
```
