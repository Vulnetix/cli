title: "VNX-619 – Dangling Database Cursor"
description: "Detect database cursors and ResultSets opened without guaranteed closure, leaking database connections."
---

## Overview

Database cursors, ResultSets, and connections that are opened but not guaranteed to be closed in all code paths (including exception paths) leak database resources. Connection pool exhaustion leads to denial of service for all application users.

**Severity:** Low | **CWE:** [CWE-619 – Dangling Database Cursor](https://cwe.mitre.org/data/definitions/619.html)

## Why This Matters

Each unclosed cursor holds a database connection slot. Under load, connection pools exhaust quickly and new requests fail. This is both a denial-of-service vulnerability and an operational reliability issue.

## What Gets Flagged

```python
# FLAGGED: cursor not using context manager
cursor = conn.cursor()
cursor.execute("SELECT * FROM users")
# If execute raises, cursor is never closed
```

```java
// FLAGGED: ResultSet without try-with-resources or finally close
ResultSet rs = stmt.executeQuery(query);
processResults(rs);
// rs never closed
```

## Remediation

Use context managers and try-with-resources:

```python
# SAFE: context manager ensures cursor is closed
with conn.cursor() as cursor:
    cursor.execute("SELECT * FROM users")
    return cursor.fetchall()
```

```java
// SAFE: try-with-resources closes ResultSet
try (ResultSet rs = stmt.executeQuery(query)) {
    return processResults(rs);
}
```
