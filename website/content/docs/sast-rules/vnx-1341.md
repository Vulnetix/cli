title: "VNX-1341 – Multiple Releases of Same Resource or Handle"
description: "Detect double-free patterns in C/C++, double-close of file handles in Python without context managers, and Go defer+close combinations that may result in two Close() calls."
---

## Overview

Releasing a resource (memory, file descriptor, mutex, socket) more than once causes undefined behaviour. In C/C++, double-free corrupts the malloc heap and is a well-known exploitation primitive for arbitrary code execution. Double-closing a file descriptor can cause a later unrelated `open()` to inherit the descriptor number, creating a confused-deputy vulnerability. **Severity:** High | **CWE:** [CWE-1341 – Multiple Releases of Same Resource or Handle](https://cwe.mitre.org/data/definitions/1341.html)

## Why This Matters

Double-free vulnerabilities have been used in high-profile exploits including CVE-2014-1266 (Apple SSL/TLS "goto fail") adjacent issues and multiple glibc-based exploits. In managed languages, double-close can cause resource handle confusion — Python's file objects raise `ValueError: I/O operation on closed file`, but in concurrent code the file may have been reassigned between closes, closing the wrong descriptor.

## What Gets Flagged

```c
// FLAGGED: free() on pointer that may be freed elsewhere
char *buf = malloc(1024);
process(buf);
free(buf);
// ... later in error path:
free(buf);  // double-free if process() raised an error and also freed
```

```python
# FLAGGED: explicit .close() without with statement
f = open('file.txt', 'r')
data = f.read()
f.close()
# ... in error handler:
f.close()  # potential double-close
```

```go
// FLAGGED: defer .Close() with also an explicit .Close()
f, err := os.Open("file.txt")
if err != nil { return err }
defer f.Close()
// ...
f.Close()  // explicit close before defer fires = double-close
```

## Remediation

```c
// SAFE: Null pointer after free to prevent double-free
char *buf = malloc(1024);
process(buf);
free(buf);
buf = NULL;  // subsequent free(NULL) is a no-op
```

```python
# SAFE: Use context manager — guarantees exactly-once close
with open('file.txt', 'r') as f:
    data = f.read()
# f.close() called automatically, exactly once
```

```go
// SAFE: Use only defer, not explicit close
f, err := os.Open("file.txt")
if err != nil { return err }
defer f.Close()  // do not call f.Close() anywhere else in this function
```
