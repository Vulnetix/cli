---
title: "VNX-259 – Use of Hard-coded Password"
description: "Detect passwords, secrets, and API keys assigned as string literals in source code across all supported languages."
---

## Overview

This rule flags source files where a password, secret, or API key variable is assigned a string literal. Hard-coded credentials become part of the compiled binary, the repository history, and any container image built from the code — all of which may be accessible to attackers who should never see production secrets. This maps to [CWE-259: Use of Hard-coded Password](https://cwe.mitre.org/data/definitions/259.html).

**Severity:** Critical | **CWE:** [CWE-259 – Use of Hard-coded Password](https://cwe.mitre.org/data/definitions/259.html)

## Why This Matters

Hard-coded passwords are one of the most common and highest-impact classes of vulnerability in enterprise software. Once an attacker obtains the binary or repository, they can extract the credential in seconds using `strings`, `grep`, or a decompiler. Unlike compromised runtime credentials that can be rotated in minutes, a hard-coded credential requires a code change, rebuild, redeploy cycle — and even then the credential exists permanently in every prior git commit and in every shipped binary. Public repository leaks (GitHub, npm, PyPI) have repeatedly exposed hard-coded passwords that granted access to production databases, cloud accounts, and payment systems. CAPEC-191 covers harvesting credentials from published sources, and MITRE ATT&CK T1552.001 (Unsecured Credentials: Credentials In Files) describes exactly this threat.

## What Gets Flagged

The rule fires on any line across any language file that contains patterns such as `password = "`, `passwd = "`, `secret = "`, `api_key = "`, `apiKey = "`, `private_key = "`, `PASSWORD = "`, `SECRET = "`, `password := "`, `Password: "`, `$password = "`, `.setPassword("`, `String password = "`, or `apiKey: "` followed by a string value.

```python
# FLAGGED: hard-coded password in Python
DATABASE_PASSWORD = "S3cr3tP@ssw0rd!"
api_key = "sk-prod-abc123def456"
```

```go
// FLAGGED: hard-coded password in Go
password := "hunter2"
cfg := Config{Password: "mysecretpassword"}
```

```java
// FLAGGED: hard-coded password in Java
String password = "admin1234";
dataSource.setPassword("db_secret_here");
```

```javascript
// FLAGGED: hard-coded API key in JavaScript/TypeScript
const apiKey = "AIzaSy-abc123xyz";
const config = { password: "supersecret", apiKey: "my-api-key" };
```

```php
// FLAGGED: hard-coded password in PHP
$password = "correcthorsebatterystaple";
$passwd = "admin";
```

## Remediation

1. **Use environment variables.** Read credentials at runtime from the process environment. Never commit `.env` files to version control.

```python
# SAFE: load from environment
import os
DATABASE_PASSWORD = os.environ["DATABASE_PASSWORD"]
api_key = os.environ["API_KEY"]
```

```go
// SAFE: load from environment
password := os.Getenv("DB_PASSWORD")
```

```java
// SAFE: load from environment
String password = System.getenv("DB_PASSWORD");
```

2. **Use a secrets manager.** For production systems, prefer a dedicated secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager, Azure Key Vault) over environment variables. Secrets managers provide audit logs, rotation, and fine-grained access control.

3. **Scan git history.** Hard-coded credentials that have been removed from the current codebase may still exist in git history. Use `git log -S "the_secret"` or a tool like `trufflehog` or `gitleaks` to scan history, then rotate any found credentials.

4. **Rotate immediately.** Treat any hard-coded credential as compromised from the moment it was committed, regardless of repository visibility.
