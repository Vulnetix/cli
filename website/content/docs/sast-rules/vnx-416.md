## Overview

A use-after-free (UAF) vulnerability occurs when code continues to read from or write to a pointer after the memory it references has been returned to the allocator. The freed memory may be reallocated for a different purpose; subsequent access reads attacker-controlled data or corrupts a newly allocated object's state. In modern heap exploits, UAF is often the primary primitive used to corrupt heap metadata, forge object pointers, and achieve type confusion leading to code execution. This maps to [CWE-416: Use After Free](https://cwe.mitre.org/data/definitions/416.html).

**Severity:** Critical | **CWE:** [CWE-416 – Use After Free](https://cwe.mitre.org/data/definitions/416.html)

## Why This Matters

Use-after-free is consistently in the top five most exploited vulnerability classes in browser engines, kernels, and network daemons. Modern allocators use tcache (glibc) or freelist metadata that is stored inline in freed chunks — an attacker who can write to freed memory can corrupt allocator metadata and achieve arbitrary allocation. The Chromium and Firefox security teams classify UAF as a critical-severity class. The pattern is also a common root cause in C++ codebases where raw pointers are used alongside RAII objects, or where shared ownership is implemented manually.

## What Gets Flagged

The rule flags every call to `free(`, `delete `, or `delete[]` in C/C++ files. Each site is worth reviewing because the pointer should be nulled immediately after freeing, and all code paths that might access the pointer after this point should be audited.

```c
/* FLAGGED: ptr is freed — is it accessed afterwards? */
free(ptr);
// ... later code that might dereference ptr ...
```

```cpp
/* FLAGGED: raw delete — ptr may be accessible via another reference */
delete obj;
obj->method();  // use after free — undefined behaviour
```

## Remediation

```c
/* SAFE: null the pointer immediately after freeing */
free(ptr);
ptr = NULL;
/* Any subsequent dereference of NULL will crash immediately (fail-safe) */

/* SAFE: use a wrapper macro */
#define SAFE_FREE(p) do { free(p); (p) = NULL; } while (0)
SAFE_FREE(ptr);
```

In C++, prefer RAII and smart pointers over manual `delete`:

```cpp
#include <memory>

// SAFE: unique_ptr automatically frees on scope exit
std::unique_ptr<MyObject> obj = std::make_unique<MyObject>();
obj->method();
// obj freed here — no dangling pointer possible

// SAFE: shared_ptr for shared ownership
std::shared_ptr<MyObject> shared = std::make_shared<MyObject>();
std::weak_ptr<MyObject> weak = shared;
// weak.lock() returns nullptr if object has been freed
```

Enable AddressSanitizer (`-fsanitize=address`) and Valgrind to detect use-after-free at runtime during testing.
