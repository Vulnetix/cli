## Overview

CWE-699 is a parent category for software development weaknesses. This rule flags security-relevant `TODO`/`FIXME`/`HACK` comments that indicate unresolved security issues, and commented-out credential or secret assignments that leave sensitive values in version history. **Severity:** Info | **CWE:** [CWE-699 – Software Development](https://cwe.mitre.org/data/definitions/699.html)

## Why This Matters

TODO comments relating to security rarely get fixed before shipping. Commented-out passwords or API keys are preserved forever in git history even after deletion. Automated secret scanning tools routinely find credentials in comments. The presence of these patterns is a reliable indicator that security debt is being deferred.

## What Gets Flagged

```python
# FLAGGED: security-related TODO
# TODO: validate user input here before processing
# FIXME: hardcoded API key below

# FLAGGED: commented-out credential
# password = "supersecret"
# API_KEY = "sk-abc123"
```

```javascript
// FLAGGED: commented-out secret
// SECRET_KEY = "my-secret-key"

// FLAGGED: security note left in code
// TODO: fix security vulnerability in auth check
```

## Remediation

1. **Resolve all security TODOs before merging.** Add a CI gate that fails builds containing `TODO: security` or `FIXME: security`.

2. **Remove commented-out secrets entirely.** Use `git filter-repo` to purge them from history if they were ever committed.

3. **Use a secrets manager.** Replace hardcoded values with calls to AWS Secrets Manager, HashiCorp Vault, or environment variables.
