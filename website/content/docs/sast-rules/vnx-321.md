---
title: "VNX-321 – Use of Hard-coded Cryptographic Key"
description: "Detect AES, RSA, HMAC, and other cryptographic keys embedded as string literals or PEM blocks in source code."
---

## Overview

This rule flags source files where a cryptographic key — AES, RSA, HMAC, signing key, or private key — is assigned a literal value, or where a PEM-encoded private key block (`-----BEGIN RSA PRIVATE KEY-----` etc.) appears outside of a dedicated key file. Hard-coded keys render the associated cryptographic operation effectively useless for confidentiality or integrity, because the key is available to anyone with access to the binary or repository. This maps to [CWE-321: Use of Hard-coded Cryptographic Key](https://cwe.mitre.org/data/definitions/321.html).

**Severity:** Critical | **CWE:** [CWE-321 – Use of Hard-coded Cryptographic Key](https://cwe.mitre.org/data/definitions/321.html)

## Why This Matters

The security of a symmetric cipher or HMAC depends entirely on the secrecy of the key, not the secrecy of the algorithm. A 256-bit AES key found in source code provides no more security than a Caesar cipher, because the key is the entire secret. Similarly, an RSA private key embedded in a source file can be extracted by any party with read access to the repository, its git history, build artifacts, or container images. Rotating a hard-coded key requires a code change and redeploy, leaving a window during which the old key is still in git history. Real-world incidents include signed Android APKs with the signing key in the repository, JWT libraries shipped with known test keys used in production, and encrypted database backups whose decryption key was in the same repository.

## What Gets Flagged

The rule fires on variable assignments with patterns such as `aesKey = "`, `hmacKey = "`, `privateKey = "`, `signingKey = "`, `secretKey = "`, `encryptionKey = "`, `KEY = "`, `AES_KEY = "`, `HMAC_KEY = "`, and equivalents in Go (`:=`), Java (`String`), Python, PHP (`$`), JavaScript/TypeScript, and Ruby. It also fires on any occurrence of PEM private key headers (`-----BEGIN RSA PRIVATE KEY-----`, `-----BEGIN EC PRIVATE KEY-----`, `-----BEGIN PRIVATE KEY-----`, etc.) outside of dedicated `.pem`, `.key`, `.p12`, or `.pfx` files.

```python
# FLAGGED: hard-coded AES key
aes_key = "0123456789abcdef0123456789abcdef"
hmac_key = "supersecrethmackey"
```

```go
// FLAGGED: hard-coded signing key in Go
signingKey := "my-jwt-signing-secret"
```

```java
// FLAGGED: hard-coded key in Java
String aesKey = "0123456789abcdef";
```

```javascript
// FLAGGED: PEM key in JavaScript source
const privateKey = `-----BEGIN RSA PRIVATE KEY-----
MIIEpAIBAAKCAQEA0Z3...
-----END RSA PRIVATE KEY-----`;
```

## Remediation

1. **Load keys from environment variables or a KMS at startup.** Never embed key material in code.

```go
// SAFE: load key from environment
key := []byte(os.Getenv("AES_ENCRYPTION_KEY"))
if len(key) != 32 {
    log.Fatal("AES_ENCRYPTION_KEY must be 32 bytes")
}
```

2. **Use a Key Management Service (KMS).** AWS KMS, GCP Cloud KMS, Azure Key Vault, and HashiCorp Vault can generate, store, and rotate keys without the application ever seeing the raw key material.

3. **Generate keys with a CSPRNG at first run.** If a key must be generated locally, use a cryptographically secure RNG and store the result in an encrypted secrets store — not in source code.

```go
// SAFE: generate AES-256 key with crypto/rand
key := make([]byte, 32)
if _, err := rand.Read(key); err != nil {
    log.Fatal(err)
}
// Store key securely — not in the repository
```

4. **Rotate any exposed keys immediately.** Treat all keys ever committed to version control as compromised regardless of repository visibility. Scan git history with `gitleaks` or `trufflehog`.
