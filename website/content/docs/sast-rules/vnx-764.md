title: "VNX-764 – Multiple Locks of a Non-recursive Mutex"
description: "Detect double-lock patterns on non-recursive mutexes that cause deadlocks in Go, Python, and Java."
---

## Overview

This rule flags consecutive mutex lock calls without an intervening unlock. Locking a non-recursive mutex a second time from the same thread/goroutine causes a deadlock: the goroutine waits for itself to release a lock it already holds. In Go, this causes a fatal runtime panic; in Python, it hangs indefinitely; in Java with a `ReentrantLock` called `lock.lock()` twice without `lock.unlock()` it deadlocks. This maps to [CWE-764: Multiple Locks of a Non-recursive Mutex](https://cwe.mitre.org/data/definitions/764.html).

**Severity:** Medium | **CWE:** [CWE-764](https://cwe.mitre.org/data/definitions/764.html)

## Why This Matters

Deadlocks in security-sensitive code paths cause denial of service. A deadlock in an authentication handler, session manager, or payment processor makes the service permanently unavailable to affected goroutines/threads, which in the worst case requires a process restart.

## What Gets Flagged

```go
// FLAGGED: double Lock() without Unlock between
var mu sync.Mutex
func process() {
    mu.Lock()
    mu.Lock()   // deadlock! first lock never released
    defer mu.Unlock()
}
```

```python
# FLAGGED: consecutive acquire() calls
import threading
lock = threading.Lock()
lock.acquire()
lock.acquire()  # deadlock! Threading.Lock is not reentrant
```

## Remediation

```go
// SAFE: single Lock with defer Unlock
func process() {
    mu.Lock()
    defer mu.Unlock()
    // ... critical section ...
}

// SAFE: if re-entrance is needed, use sync.RWMutex or a reentrant design
```

```python
# SAFE: use with statement for automatic acquire/release
with lock:
    # critical section

# SAFE: use RLock if re-entrance is needed
rlock = threading.RLock()
with rlock:
    with rlock:  # safe: RLock is reentrant
        pass
```
