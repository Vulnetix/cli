---
title: "VNX-1188 – Initialization of a Resource with an Insecure Default"
description: "Detect insecure defaults: DEBUG=True in Django/Flask, weak SECRET_KEY values, and ALLOWED_HOSTS=['*'] left in production configurations."
---

## Overview

Development-time defaults left in production are a leading cause of security incidents. Django's `DEBUG = True` exposes full stack traces, SQL query logs, and environment variables to users. Flask's `app.run(debug=True)` activates the interactive Werkzeug debugger that allows arbitrary code execution. Wildcard `ALLOWED_HOSTS` disables Host header validation, enabling host-header injection attacks. **Severity:** High | **CWE:** [CWE-1188 – Initialization of a Resource with an Insecure Default](https://cwe.mitre.org/data/definitions/1188.html)

## Why This Matters

In 2018 the Uber breach was partially attributed to exposed debug endpoints. The Django debug toolbar reveals database queries, local variables, and request headers on every 500 error page — information that is invaluable to an attacker. Weak `SECRET_KEY` values allow session forgery and CSRF token bypass.

## What Gets Flagged

```python
# FLAGGED: Django DEBUG in production
DEBUG = True

# FLAGGED: Flask debug mode
app.run(debug=True)

# FLAGGED: Weak secret key
SECRET_KEY = 'dev'
SECRET_KEY = 'django-insecure-placeholder'

# FLAGGED: Open host validation
ALLOWED_HOSTS = ['*']
```

## Remediation

```python
# SAFE: Load from environment, default to False
import os
DEBUG = os.environ.get('DJANGO_DEBUG', 'False').lower() == 'true'

# SAFE: Strong random secret key
import secrets
SECRET_KEY = os.environ.get('SECRET_KEY') or secrets.token_hex(50)

# SAFE: Explicit host list
ALLOWED_HOSTS = os.environ.get('ALLOWED_HOSTS', 'yourdomain.com').split(',')

# SAFE: Flask — never run with debug=True in production
if __name__ == '__main__':
    app.run(debug=os.environ.get('FLASK_DEBUG', 'false').lower() == 'true')
```
