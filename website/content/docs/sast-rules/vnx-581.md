---
title: "VNX-581 – Object Model Violation: equals() Without hashCode()"
description: "Detect Java classes that define equals() without hashCode() or vice versa, breaking collection semantics."
---

## Overview

Java's `equals()` and `hashCode()` contract requires that objects which compare equal must have the same hash code. Defining one without the other causes incorrect behavior in `HashMap`, `HashSet`, and other hash-based collections, which can lead to authorization objects being missed in lookups.

**Severity:** Low | **CWE:** [CWE-581 – Object Model Violation: Just One of Equals and Hashcode Defined](https://cwe.mitre.org/data/definitions/581.html)

## Why This Matters

A `Permission` or `Role` object stored in a `HashSet` with a custom `equals()` but no `hashCode()` may never be found during a lookup, causing authorization checks to fail open. Access control lists implemented with hash collections are particularly vulnerable.

## What Gets Flagged

```java
// FLAGGED: equals() without hashCode()
public class Permission {
    @Override
    public boolean equals(Object o) { ... }
    // Missing: hashCode()
}
```

## Remediation

Always define both methods consistently:

```java
// SAFE: Both methods defined
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Permission)) return false;
    Permission p = (Permission) o;
    return Objects.equals(name, p.name);
}

@Override
public int hashCode() {
    return Objects.hash(name);
}
```
