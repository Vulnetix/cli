---
title: "VNX-733 – Compiler Optimization Removal of Security-critical Code"
description: "Detect memset() on sensitive buffers before free() that compilers may optimise away, leaving passwords and keys in memory."
---

## Overview

This rule flags `memset()` and `bzero()` calls on buffers with security-sensitive names (password, key, secret) that are immediately followed by `free()` or go out of scope. Modern optimising compilers eliminate these as dead stores — the memory is about to be freed, so writing to it appears unnecessary from the compiler's perspective. This leaves credentials in freed memory where they may remain readable. This maps to [CWE-733: Compiler Optimization Removal of Security-critical Code](https://cwe.mitre.org/data/definitions/733.html), also known as CWE-14.

**Severity:** Medium | **CWE:** [CWE-733](https://cwe.mitre.org/data/definitions/733.html)

## Why This Matters

Memory forensics tools and cold-boot attacks can read freed memory. If a password buffer is `memset()`'d to zero but the compiler removes that call, the plaintext password remains in the heap until it is overwritten by a subsequent allocation — potentially minutes or hours later. This is not a theoretical concern; it has been demonstrated in several real-world attacks.

## What Gets Flagged

```c
// FLAGGED: memset followed by free (compiler may eliminate the memset)
void login(const char *username) {
    char password_buf[256];
    read_password(password_buf, sizeof(password_buf));
    verify(username, password_buf);
    memset(password_buf, 0, sizeof(password_buf));  // may be optimised away
    free(password_buf);
}
```

```c
// FLAGGED: bzero on key buffer
bzero(secret_key, key_len);
free(secret_key);
```

## Remediation

Use functions that compilers are required to honour:

```c
// SAFE: C11 memset_s (guaranteed not optimised away)
#include <string.h>
memset_s(password_buf, sizeof(password_buf), 0, sizeof(password_buf));

// SAFE: explicit_bzero (POSIX, available on Linux/macOS)
#include <strings.h>
explicit_bzero(password_buf, sizeof(password_buf));

// SAFE: SecureZeroMemory on Windows
SecureZeroMemory(password_buf, sizeof(password_buf));
```

On platforms without these, use a volatile write loop that the compiler cannot legally eliminate:

```c
volatile unsigned char *p = (volatile unsigned char *)password_buf;
size_t n = sizeof(password_buf);
while (n--) *p++ = 0;
```
