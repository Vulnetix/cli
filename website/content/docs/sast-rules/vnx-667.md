---
title: "VNX-667 – Improper Locking"
description: "Detect mutex and lock acquisition in Go, Python, and Java that lacks a guaranteed paired release, which can cause deadlocks or unprotected concurrent access."
---

## Overview

Proper locking requires that every acquisition is paired with a guaranteed release regardless of control flow. Missing the paired unlock, or unlocking on only some code paths, leaves either a deadlock (the lock is held forever) or a window of unprotected concurrent access (if the unlock happens before it should). CWE-667 covers both over-locking (deadlock) and under-locking (data race). In Go the idiomatic fix is `defer mu.Unlock()` immediately after `mu.Lock()`. In Python, `with lock:` ensures release even on exception. In Java, unlock must be in a `finally` block.

**Severity:** Medium | **CWE:** [CWE-667 – Improper Locking](https://cwe.mitre.org/data/definitions/667.html)

## Why This Matters

A missed unlock causes a goroutine or thread to block indefinitely the next time it tries to acquire the same lock — a deadlock that typically requires a service restart to recover from. In availability-critical systems this is a denial-of-service vulnerability. Conversely, if a lock is released too early due to an exception unwinding the stack, the protected data is exposed to concurrent access with no synchronisation, producing data races that can corrupt state silently. Both problems are hard to reproduce in tests because they depend on scheduling timing.

## What Gets Flagged

```go
// FLAGGED: Lock() without immediate defer Unlock()
mu.Lock()
// ... code that may panic or return early ...
mu.Unlock()  // not reached if panic occurs
```

```python
# FLAGGED: acquire() without context manager — release not guaranteed
lock.acquire()
do_work()       # if this raises, release() is never called
lock.release()
```

```java
// FLAGGED: lock() without finally { unlock() }
lock.lock();
doSomething();   // if this throws, lock is never released
lock.unlock();
```

## Remediation

```go
// SAFE: defer guarantees Unlock runs even on panic or early return
mu.Lock()
defer mu.Unlock()
// ... protected code ...
```

```python
# SAFE: context manager guarantees release on any exit
with lock:
    do_work()
```

```java
// SAFE: finally block guarantees unlock
lock.lock();
try {
    doSomething();
} finally {
    lock.unlock();
}
```

For `sync.RWMutex` in Go, the same pattern applies to `RLock`/`RUnlock`:

```go
mu.RLock()
defer mu.RUnlock()
value := sharedMap[key]
```

Use Go's race detector (`go test -race`) and Java's thread sanitizers to detect lock misuse at runtime.
