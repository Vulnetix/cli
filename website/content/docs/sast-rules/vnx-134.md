---
title: "VNX-134 – Use of Externally-Controlled Format String"
description: "Detect format string injection in C/C++, Go, and Python where user-controlled data is passed directly as the format argument to printf-family functions or logging calls."
---

## Overview

Format string vulnerabilities occur when user-controlled data is passed as the format string argument to `printf`, `fprintf`, `sprintf` (C), `fmt.Printf` (Go), or logging functions (Python), rather than as a data argument. The format string is a program instruction: `%n` writes to memory, `%s` reads from the stack, and `%x` leaks stack bytes. An attacker who controls the format string can read arbitrary process memory, write to arbitrary addresses, and in classic C implementations achieve arbitrary code execution. In Go and Python the direct code-execution risk is lower but information disclosure and crashes are readily achievable.

**Severity:** High | **CWE:** [CWE-134 – Use of Externally-Controlled Format String](https://cwe.mitre.org/data/definitions/134.html)

## Why This Matters

Format string bugs fell out of fashion after the mid-2000s but resurface regularly in embedded firmware, IoT devices, and code that calls `syslog` or `printf` with user-supplied messages. The `%n` specifier writes the number of bytes printed so far to an int pointer — effectively an arbitrary memory write primitive. Even without `%n`, repeated `%x` or `%p` specifiers allow stack dumping to exfiltrate secrets, canary values, and addresses that defeat ASLR. In interpreted languages the risk is typically DoS or information disclosure rather than code execution, but both are serious.

## What Gets Flagged

```c
/* FLAGGED: user input as the format string — %n allows memory write */
printf(argv[1]);
fprintf(stderr, user_message);
sprintf(buf, user_input);
syslog(LOG_INFO, user_log_message);
```

```go
// FLAGGED: request-derived value used as format string
fmt.Printf(r.FormValue("msg"))
fmt.Fprintf(w, r.URL.Path)
log.Printf(req.Header.Get("X-Log"))
```

```python
# FLAGGED: user input passed directly to logging (format injection)
logging.info(user_input)
logging.error(request.GET['msg'])
```

## Remediation

Always supply a literal format string and pass user data as arguments:

```c
/* SAFE: literal format string, user data as argument */
printf("%s", argv[1]);
fprintf(stderr, "%s", user_message);
snprintf(buf, sizeof(buf), "%s", user_input);
syslog(LOG_INFO, "%s", user_log_message);
```

```go
// SAFE: explicit format verb, user value as argument
fmt.Printf("%s", r.FormValue("msg"))
fmt.Fprintf(w, "%s", r.URL.Path)
log.Printf("%s", req.Header.Get("X-Log"))

// Or use the non-format variants:
fmt.Fprint(w, r.URL.Path)
log.Print(req.Header.Get("X-Log"))
```

```python
# SAFE: use % placeholder or structured logging
logging.info("%s", user_input)
logging.error("Request message: %s", request.GET['msg'])
```

Enable GCC's `-Wformat -Wformat-security` warnings to catch format string misuse at compile time in C/C++.
