---
title: "VNX-167 – Improper Handling of Additional Special Element"
description: "Detect file operations using user input without stripping null bytes, which can truncate filenames and bypass security checks."
---

## Overview

This rule flags file operations (`fopen`, `file_get_contents`, `open()`) that use user-supplied paths without first stripping null bytes (`\0`). A null byte embedded in a filename truncates it at the C layer while the full string is visible to PHP/Python validation. An attacker requests `/etc/passwd\0.jpg` which passes an extension check but opens `/etc/passwd`. This maps to [CWE-167: Improper Handling of Additional Special Element](https://cwe.mitre.org/data/definitions/167.html).

**Severity:** Medium | **CWE:** [CWE-167 – Improper Handling of Additional Special Element](https://cwe.mitre.org/data/definitions/167.html)

## What Gets Flagged

```php
<?php
// FLAGGED: fopen with user-supplied path (may contain null bytes)
$filename = $_GET['file'];
$fh = fopen($filename, 'r');  // $_GET['file'] could be "secret\0.txt"
```

```python
# FLAGGED: open with user path without null byte check
filename = request.args['file']
with open(os.path.join(base, filename)) as f:
    return f.read()
```

## Remediation

```php
<?php
// SAFE: strip null bytes
$filename = str_replace("\0", '', $_GET['file']);
$filename = basename($filename);  // also prevent path traversal
if (pathinfo($filename, PATHINFO_EXTENSION) !== 'txt') {
    die('Invalid file type');
}
$fh = fopen('/safe/dir/' . $filename, 'r');
```

```python
# SAFE: remove null bytes
filename = request.args['file'].replace('\x00', '')
safe_path = os.path.join(base, filename)
real_path = os.path.realpath(safe_path)
if not real_path.startswith(os.path.realpath(base)):
    abort(403)
```
