---
title: "VNX-821 – Incorrect Synchronization"
description: "Detect synchronization on string literals, boxed primitives, or non-final lock fields that provide no mutual exclusion guarantee."
---

## Overview

This rule flags synchronization constructs that appear correct but provide no actual thread safety: locking on string literals (which may be interned and shared across the JVM), locking on `Integer`/`Boolean` autoboxed values (which are cached and may be modified), and lock objects declared as non-`final` fields that can be replaced. This maps to [CWE-821: Incorrect Synchronization](https://cwe.mitre.org/data/definitions/821.html).

**Severity:** Medium | **CWE:** [CWE-821](https://cwe.mitre.org/data/definitions/821.html)

## Why This Matters

`synchronized ("lock")` in Java locks on an interned String object shared across the entire JVM. Two completely unrelated classes synchronizing on `"lock"` contend for the same monitor. `synchronized (integerField)` with an `Integer` object will lock on different objects as the value changes due to autoboxing, providing no mutual exclusion. Non-final lock fields can be reassigned, causing threads to synchronize on different monitors.

## What Gets Flagged

```java
// FLAGGED: synchronizing on string literal — shared intern pool
synchronized ("myLock") {
    criticalSection();
}

// FLAGGED: synchronizing on boxed primitive
Integer count = 0;
synchronized (count) {  // count is reassigned during autoboxing increment
    count++;
}

// FLAGGED: synchronizing on Boolean
Boolean flag = Boolean.TRUE;
synchronized (flag) {  // Boolean.TRUE is a shared constant
    flag = !flag;
}
```

```java
// FLAGGED: non-final lock object that could be reassigned
private Object lock = new Object();
// ... elsewhere:
lock = new Object();  // now synchronized (lock) uses a different monitor
```

```go
// FLAGGED: value receiver on method that uses Mutex — copies the lock
type Counter struct {
    mu sync.Mutex
    n  int
}

func (c Counter) Increment() {  // value receiver copies c.mu — broken
    c.mu.Lock()
    c.n++
    c.mu.Unlock()
}
```

## Remediation

```java
// SAFE: private final dedicated lock object
private final Object lock = new Object();

synchronized (lock) {
    criticalSection();
}

// SAFE: use java.util.concurrent locks
import java.util.concurrent.locks.ReentrantLock;
private final ReentrantLock lock = new ReentrantLock();

lock.lock();
try {
    criticalSection();
} finally {
    lock.unlock();
}

// SAFE: AtomicInteger for counter
import java.util.concurrent.atomic.AtomicInteger;
private final AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();
```

```go
// SAFE: pointer receiver preserves the same mutex
func (c *Counter) Increment() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.n++
}
```
