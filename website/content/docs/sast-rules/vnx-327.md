## Overview

This rule flags the use of cryptographic algorithms that are known to be broken or insufficiently secure for modern applications: MD5, SHA-1, DES (56-bit key), Triple DES (Sweet32 vulnerability), RC4 (statistical biases), and Blowfish (small block size). These algorithms are found across all supported languages through their standard library imports and API call sites. This maps to [CWE-327: Use of a Broken or Risky Cryptographic Algorithm](https://cwe.mitre.org/data/definitions/327.html).

**Severity:** High | **CWE:** [CWE-327 – Use of a Broken or Risky Cryptographic Algorithm](https://cwe.mitre.org/data/definitions/327.html)

## Why This Matters

Algorithm selection is a long-lived decision. Data encrypted today with a weak algorithm may need to remain confidential for years, but the algorithm that was acceptable in 2000 is now breakable with commodity cloud compute. DES was publicly cracked in 22 hours in 1999 with a $250,000 machine; that cost is now a few dollars of spot compute. Triple DES was deprecated by NIST in 2017 (SP 800-131A) and prohibited after 2023. RC4 was banned in TLS by RFC 7465 after practical attacks were demonstrated against WEP and SSL. MD5 and SHA-1 have known collision attacks: identical digests can be produced for distinct inputs (Google's SHAttered, chosen-prefix attacks). Attackers exploit these weaknesses to forge certificates, decrypt stored data, bypass integrity checks, and intercept communications. MITRE ATT&CK T1600 explicitly covers weakening cryptography as a technique used to facilitate further compromise.

## What Gets Flagged

The rule fires on language-specific patterns including:

- **Python**: `hashlib.md5(`, `hashlib.sha1(`, `Crypto.Cipher.DES`, `Crypto.Cipher.Blowfish`, `Crypto.Cipher.ARC4`
- **Java**: `MessageDigest.getInstance("MD5")`, `MessageDigest.getInstance("SHA-1")`, `Cipher.getInstance("DES`, `Cipher.getInstance("RC4")`, `Cipher.getInstance("Blowfish`
- **JavaScript/Node.js**: `crypto.createHash('md5')`, `crypto.createHash('sha1')`, `crypto.createCipheriv('des`, `crypto.createCipheriv('rc4`
- **PHP**: `mcrypt_encrypt(`, `mcrypt_decrypt(`
- **Ruby**: `Digest::MD5.`, `Digest::SHA1.`, `OpenSSL::Cipher.new("DES`
- **Go**: `"crypto/md5"`, `"crypto/sha1"`, `"crypto/des"`, `"crypto/rc4"`, `md5.New(`, `sha1.New(`, `des.NewCipher(`, `rc4.NewCipher(`

```python

```

```go
// FLAGGED: DES cipher in Go

```

```java

```

```javascript
// FLAGGED: MD5 hash in Node.js

```

## Remediation

1. **For encryption**, replace DES/3DES/RC4/Blowfish with **AES-256-GCM**. Use `crypto/aes` + `cipher.NewGCM` in Go, `javax.crypto.Cipher` with `"AES/GCM/NoPadding"` in Java, `crypto.createCipheriv('aes-256-gcm', ...)` in Node.js, `cryptography.hazmat.primitives.ciphers.algorithms.AES` in Python.

2. **For integrity/checksums of non-password data**, replace MD5/SHA-1 with **SHA-256** or **SHA-3**.

```go
// SAFE: SHA-256 in Go
import "crypto/sha256"
hash := sha256.Sum256(data)
```

```python
# SAFE: SHA-256 in Python
import hashlib
digest = hashlib.sha256(data).hexdigest()
```

3. **For password hashing**, use an adaptive algorithm: `bcrypt`, `argon2`, or `scrypt`. Never use MD5, SHA-1, or SHA-256 directly for passwords — they are too fast and enable brute-force.

4. **Re-encrypt data at rest** that was protected with broken algorithms. Treat all data encrypted with DES, 3DES, or RC4 as potentially compromised.
