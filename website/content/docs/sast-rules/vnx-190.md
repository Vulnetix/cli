title: "VNX-190 – Integer Overflow or Wraparound"
description: "Detect integer arithmetic used to compute allocation sizes or array indices without overflow checking, in C/C++, Java, and Go."
---

## Overview

When an integer used to compute a buffer size or array index overflows its type, the resulting value is far smaller than intended. The allocation succeeds — because the small size is a valid request — but the code then writes the full intended amount of data, overflowing the undersized buffer. In C's `size_t`, a multiplication `count * item_size` wraps at `SIZE_MAX + 1`. In Java, `int` arithmetic wraps at `Integer.MAX_VALUE`. In Go, `int` on a 32-bit platform is 32 bits and can wrap; on 64-bit platforms the risk is lower but not eliminated when values come from network or user input.

**Severity:** High | **CWE:** [CWE-190 – Integer Overflow or Wraparound](https://cwe.mitre.org/data/definitions/190.html)

## Why This Matters

Integer overflow vulnerabilities are notoriously subtle: the overflow happens silently, the allocation appears to succeed, and the crash or corruption occurs later and may look unrelated. They are frequently used in heap exploitation chains — the attacker supplies a crafted size that overflows to a small value, triggering an underallocation, then exploits the subsequent heap overflow. This pattern appears repeatedly in real-world CVEs including image processing libraries (libjpeg, libpng) and archive extractors (zlib, libarchive). On 32-bit embedded targets the risk is especially high because `size_t` is only 32 bits.

## What Gets Flagged

```c
/* FLAGGED: multiplication may overflow size_t on 32-bit targets */
char *buf = malloc(count * item_size);

/* FLAGGED: alloca with computed size — stack overflow on large values */
int *tmp = alloca(n * sizeof(int));
```

```java
// FLAGGED: allocation size from user input without bounds check
byte[] data = new byte[Integer.parseInt(request.getParameter("size"))];
```

```go
// FLAGGED: make with user-supplied size without cap
buf := make([]byte, userSize)
```

## Remediation

```c
#include <stdint.h>
#include <stdlib.h>

/* SAFE: use calloc — it performs overflow-safe multiplication internally */
char *buf = calloc(count, item_size);

/* SAFE: explicit overflow check before malloc */
if (count > SIZE_MAX / item_size) {
    return ERROR_OVERFLOW;
}
char *buf2 = malloc(count * item_size);

/* SAFE: __builtin_mul_overflow (GCC/Clang) */
size_t total;
if (__builtin_mul_overflow(count, item_size, &total)) {
    return ERROR_OVERFLOW;
}
char *buf3 = malloc(total);
```

```java
// SAFE: validate before allocation
int size = Integer.parseInt(request.getParameter("size"));
if (size < 0 || size > MAX_ALLOWED_SIZE) {
    throw new IllegalArgumentException("Size out of range: " + size);
}
byte[] data = new byte[size];
```

```go
// SAFE: cap to a reasonable maximum
const maxBufSize = 64 * 1024 * 1024 // 64 MiB
if userSize < 0 || userSize > maxBufSize {
    return nil, fmt.Errorf("requested size %d exceeds limit %d", userSize, maxBufSize)
}
buf := make([]byte, userSize)
```
