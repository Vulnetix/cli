## Overview

When an application uses user-supplied input as a filename or resource name without proper sanitization, attackers can supply names containing `../` (path traversal), null bytes, or special characters to access files outside the intended directory, overwrite system files, or trigger unexpected behavior.

**Severity:** Low | **CWE:** [CWE-641 – Improper Restriction of Names for Files and Other Resources](https://cwe.mitre.org/data/definitions/641.html)

## Why This Matters

`open(os.path.join('/uploads', request.GET['filename']))` can be exploited with `filename=../../../etc/passwd` to read arbitrary files. Even when application logic seems to restrict access, improper sanitization of the filename itself allows attackers to escape the intended directory.

## What Gets Flagged

```python
# FLAGGED: Filename directly from query parameter
filename = request.GET['filename']
with open(os.path.join(base_dir, filename)) as f:
    return f.read()

# FLAGGED: User-controlled path in file join
path = os.path.join(upload_dir, request.POST.get('name'))
```

```php
<?php
// FLAGGED: Direct use of GET parameter as filename
$filename = $_GET['file'];
readfile('/var/uploads/' . $filename);
```

## Remediation

Sanitize filenames to only allow safe characters and verify the resolved path stays within the allowed directory:

```python
import os, re

# SAFE: allowlist characters, use basename, verify realpath
def safe_open(base_dir, user_filename):
    # Strip unsafe characters
    safe_name = re.sub(r'[^a-zA-Z0-9._\-]', '', user_filename)
    safe_name = os.path.basename(safe_name)  # strip any remaining path components
    
    full_path = os.path.realpath(os.path.join(base_dir, safe_name))
    canonical_base = os.path.realpath(base_dir)
    
    if not full_path.startswith(canonical_base + os.sep):
        raise ValueError("Path traversal detected")
    
    return open(full_path)
```
