## Overview

This rule flags denylist-based input filtering that removes specific strings (e.g., `<script>`) without accounting for case variations (`<SCRIPT>`, `<Script>`), whitespace insertion (`<scr ipt>`), or nested patterns (`<scr<script>ipt>`). Attackers routinely bypass such filters. This maps to [CWE-790: Improper Filtering of Special Elements](https://cwe.mitre.org/data/definitions/790.html).

**Severity:** Medium | **CWE:** [CWE-790](https://cwe.mitre.org/data/definitions/790.html)

## Why This Matters

Any denylist-based approach to XSS prevention is fundamentally incomplete. HTML parsers in browsers are extremely permissive and will interpret variations that bypass string replacement. The correct approach is to use a context-aware output encoding library or an allowlist-based sanitizer.

## What Gets Flagged

```python
# FLAGGED: simple string replace — bypassed by <SCRIPT>
clean = user_input.replace('<script>', '')
clean = user_input.replace('javascript:', '')
```

```javascript
// FLAGGED: regex replace without case-insensitive flag
let clean = input.replace(/<script>/, '');  // <SCRIPT> passes
let clean = input.replace(/javascript:/, '');  // JAVASCRIPT: passes
```

```php
// FLAGGED: str_replace of specific tag
$clean = str_replace('<script>', '', $input);  // easily bypassed
```

## Remediation

```python
# SAFE: use a vetted HTML sanitization library
import bleach
ALLOWED_TAGS = ['b', 'i', 'u', 'strong', 'em']
clean = bleach.clean(user_input, tags=ALLOWED_TAGS, strip=True)
```

```javascript
// SAFE: DOMPurify for client-side, sanitize-html for server-side
const DOMPurify = require('dompurify');
const clean = DOMPurify.sanitize(userInput);
```

```php
// SAFE: HTMLPurifier
require_once '/path/to/HTMLPurifier.auto.php';
$purifier = new HTMLPurifier();
$clean = $purifier->purify($dirty_html);
```

For plain text output: use context-aware HTML escaping instead of allowing any HTML.
