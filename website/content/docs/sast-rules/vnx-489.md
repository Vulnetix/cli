---
title: "VNX-489 – Active Debug Code"
description: "Detect debug settings and development-mode configuration left active in production code, which exposes internal details to attackers and disables security controls."
---

## Overview

Debug mode settings left enabled in production expose detailed stack traces, internal configuration, database queries, and error messages to anyone who triggers an exception. This information significantly accelerates attacks by revealing framework versions, file paths, database schema, and application internals. Some debug settings also disable security controls such as CSRF protection and content-type validation. This maps to [CWE-489: Active Debug Code](https://cwe.mitre.org/data/definitions/489.html).

**Severity:** Medium | **CWE:** [CWE-489 – Active Debug Code](https://cwe.mitre.org/data/definitions/489.html)

## Why This Matters

Debug information disclosure accelerates every other type of attack. A stack trace showing the full file path and line number of a SQL error confirms injection points. The Django debug page shows all environment variables (including `SECRET_KEY`, database passwords, and API keys) when an unhandled exception occurs. PHP with `display_errors = On` prints full file paths and variable dumps. The Werkzeug debugger (triggered by Flask's `debug=True`) includes an interactive Python console that requires no authentication, giving anyone who can trigger an exception full RCE. MITRE ATT&CK T1082 covers system information discovery, which debug output directly enables.

## What Gets Flagged

```python
# FLAGGED: Django DEBUG = True in settings
# settings.py
DEBUG = True  # exposes full stack traces and environment variables

# FLAGGED: Flask debug mode — includes unauthenticated Werkzeug console
from flask import Flask
app = Flask(__name__)
app.run(debug=True)     # anyone triggering an exception gets Python REPL
app.run(debug = True)   # same with spaces
```

```php
<?php
// FLAGGED: PHP displaying all errors to users
error_reporting(E_ALL);
ini_set('display_errors', 'On');
// or in php.ini:
// display_errors = On
```

```yaml
# FLAGGED: Spring Boot development profile in production config
# application.properties or application.yml
spring.profiles.active=dev
spring.profiles.active=development
```

```ruby
# FLAGGED: Rails debug log level in production config
# config/environments/production.rb
config.log_level = :debug  # logs all SQL queries and parameters
config.consider_all_requests_local = true  # shows debug info to all users
config.action_controller.perform_caching = false  # disables caching in prod
```

```php
<?php
// FLAGGED: WordPress debug mode
define('WP_DEBUG', true);
```

## Remediation

### Python / Django

```python
# settings/production.py — SAFE production settings
DEBUG = False

# SAFE: Use environment variable to prevent accidental debug mode
import os
DEBUG = os.environ.get('DJANGO_DEBUG', 'False').lower() == 'true'

# SAFE: Configure error logging to a file/service, not the HTTP response
LOGGING = {
    'version': 1,
    'disable_existing_loggers': False,
    'handlers': {
        'file': {
            'level': 'ERROR',
            'class': 'logging.FileHandler',
            'filename': '/var/log/myapp/django.log',
        },
    },
    'loggers': {
        'django': {
            'handlers': ['file'],
            'level': 'ERROR',
            'propagate': True,
        },
    },
}

# SAFE: Send errors to Sentry/Datadog without exposing them to users
import sentry_sdk
sentry_sdk.init(dsn=os.environ['SENTRY_DSN'])
```

```python
# SAFE: Flask production configuration
from flask import Flask
app = Flask(__name__)

# Never call app.run() in production — use a WSGI server (gunicorn, uWSGI)
# gunicorn --bind 0.0.0.0:8000 myapp:app
```

### PHP

```php
<?php
// SAFE: Production PHP error handling — log errors, don't display them
error_reporting(E_ALL);           // capture everything
ini_set('display_errors', '0');   // never show to users
ini_set('log_errors', '1');       // always log
ini_set('error_log', '/var/log/php/error.log');

// SAFE: php.ini production settings
// display_errors = Off
// log_errors = On
// error_log = /var/log/php_errors.log
```

### Spring Boot

```yaml
# application-production.yml — SAFE production settings
spring:
  profiles:
    active: production

logging:
  level:
    root: WARN
    com.myapp: INFO
  file:
    name: /var/log/myapp/app.log
```

### Ruby on Rails

```ruby
# config/environments/production.rb — SAFE
Rails.application.configure do
  config.log_level = :warn       # not :debug
  config.consider_all_requests_local = false  # hide debug info
  config.action_controller.perform_caching = true  # enable caching
end
```

### WordPress

```php
<?php
// SAFE: wp-config.php production settings
define('WP_DEBUG', false);
define('WP_DEBUG_LOG', false);
define('WP_DEBUG_DISPLAY', false);
```
