## Overview

Detects `delete this` in C++ code. An object that destroys itself creates a use-after-free hazard: any subsequent access to member variables, virtual function calls, or access through any pointer that was holding the object constitutes undefined behaviour exploitable for arbitrary code execution.

**Severity:** Medium | **CWE:** [CWE-1082 – Class Instance Self Destruction Control Element](https://cwe.mitre.org/data/definitions/1082.html)

## Why This Matters

Use-after-free is a critical memory safety vulnerability. After `delete this`, the memory is freed, but the `this` pointer still exists. Code that continues executing in the same scope (or holds a reference) can write to freed memory, corrupting heap metadata or control flow pointers.

## What Gets Flagged

```cpp
// FLAGGED: object destroying itself
void SecurityToken::expire() {
    this->invalidate();
    delete this;  // this pointer is now dangling
    // any member access after this is UB
}
```

## Remediation

Use `std::shared_ptr` for lifetime management instead of manual deletion:

```cpp
class SecurityToken : public std::enable_shared_from_this<SecurityToken> {
    void expire() {
        this->invalidate();
        // lifetime managed by shared_ptr - no manual delete needed
    }
};
```
