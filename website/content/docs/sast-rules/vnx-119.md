title: "VNX-119 – Improper Restriction of Buffer Operations"
description: "Detect C/C++ code using unsafe string and I/O functions that do not enforce buffer boundaries, a leading cause of exploitable memory corruption."
---

## Overview

This rule flags C and C++ source files that call functions known to copy, append, or read data into fixed-size buffers without enforcing a size limit: `strcpy`, `strcat`, `sprintf`, `gets`, and `scanf` with an unguarded `%s` format specifier. These functions were part of the original C standard library and remain common in legacy and embedded codebases, yet they have no mechanism to stop writing past the end of a destination buffer. When attacker-controlled data is longer than the buffer, the excess bytes overwrite adjacent memory — the stack return address, heap metadata, or adjacent objects — enabling arbitrary code execution or denial of service.

**Severity:** High | **CWE:** [CWE-119 – Improper Restriction of Operations within the Bounds of a Memory Buffer](https://cwe.mitre.org/data/definitions/119.html)

## Why This Matters

Buffer overflows have been the single most exploited vulnerability class for over three decades. The Morris Worm (1988) used a `gets` overflow; the WannaCry ransomware exploited an SMB buffer overflow; hundreds of CVEs every year trace back to `strcpy` and `sprintf` in C codebases. Modern mitigations (ASLR, stack canaries, NX bits) raise the bar but do not eliminate exploitability — especially in embedded and real-time systems where these protections are often absent. MITRE ATT&CK T1203 (Exploitation for Client Execution) directly corresponds to buffer overflow exploitation as an initial access technique.

## What Gets Flagged

Any `.c`, `.cpp`, `.cc`, `.cxx`, `.h`, or `.hpp` file containing the following function calls:

- `strcpy(` — unbounded string copy
- `strcat(` — unbounded string concatenation
- `gets(` — reads until newline with no length limit (removed from C11)
- `sprintf(` — unbounded formatted string write
- `scanf("%s"` — reads whitespace-delimited token with no width specifier

```c
/* FLAGGED: strcpy has no knowledge of dest buffer size */
char dest[64];
strcpy(dest, argv[1]);  // argv[1] can be arbitrarily long

/* FLAGGED: strcat does not check remaining capacity */
char buf[128] = "Hello, ";
strcat(buf, username);  // username could overflow buf

/* FLAGGED: gets is unconditionally dangerous — removed in C11 */
char line[256];
gets(line);

/* FLAGGED: sprintf writes formatted output with no dest size guard */
char msg[64];
sprintf(msg, "User: %s logged in", username);

/* FLAGGED: %s with no width limit reads until whitespace */
char name[32];
scanf("%s", name);
```

## Remediation

Replace each unsafe function with a size-bounded equivalent:

```c
#include <string.h>
#include <stdio.h>

/* SAFE: strlcpy (BSD/POSIX) always null-terminates and returns required length */
char dest[64];
strlcpy(dest, argv[1], sizeof(dest));

/* SAFE: strlcat appends safely and returns the total required length */
char buf[128] = "Hello, ";
strlcat(buf, username, sizeof(buf));

/* SAFE: fgets reads at most n-1 characters and null-terminates */
char line[256];
if (fgets(line, sizeof(line), stdin) == NULL) { /* handle error */ }
/* Strip trailing newline if needed */
line[strcspn(line, "\n")] = '\0';

/* SAFE: snprintf limits output to n-1 characters */
char msg[64];
snprintf(msg, sizeof(msg), "User: %s logged in", username);

/* SAFE: width specifier limits input to buffer size - 1 */
char name[32];
scanf("%31s", name);
```

Where `strlcpy`/`strlcat` are not available (glibc does not include them by default), use `strncpy` + `strncat` with care, always ensuring explicit null termination, or adopt the Microsoft Annex K functions (`strcpy_s`, `strcat_s`) if targeting Windows.

**Compiler hardening flags** provide a safety net but are not a replacement for fixing the code:
- GCC/Clang: `-D_FORTIFY_SOURCE=2 -O2` enables compile-time and runtime buffer overflow detection
- Enable AddressSanitizer during testing: `-fsanitize=address`
