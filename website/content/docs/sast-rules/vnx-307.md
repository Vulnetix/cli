---
title: "VNX-307 – Improper Restriction of Excessive Authentication Attempts"
description: "Detect login and authentication routes defined without rate limiting, account lockout, or throttling controls."
---

## Overview

This rule flags login and authentication endpoint definitions in Python (Flask/Django), Node.js (Express), Go, and Java where no rate limiting, account lockout, or throttling indicator is present in the same file. Without these controls, an attacker can submit an unlimited number of password guesses against any account. This maps to [CWE-307: Improper Restriction of Excessive Authentication Attempts](https://cwe.mitre.org/data/definitions/307.html).

**Severity:** Medium | **CWE:** [CWE-307 – Improper Restriction of Excessive Authentication Attempts](https://cwe.mitre.org/data/definitions/307.html)

## Why This Matters

Credential-stuffing and brute-force attacks are among the most common attack vectors in web application compromise. Attackers use lists of username/password pairs leaked from other breaches (credential stuffing) or systematically try every possible password combination (brute force). Modern botnet infrastructure can distribute attacks across thousands of IP addresses, defeating simple IP-based rate limiting. Without application-level rate limiting per account, account lockout after N failures, or CAPTCHA, an attacker can test millions of credential pairs against a login endpoint within hours. The 2022 Okta breach, multiple high-profile gaming platform breaches, and countless SaaS compromises have been enabled partly by the absence of effective brute-force protection.

## What Gets Flagged

The rule detects route definitions for login/auth endpoints:

```python
@app.route("/login", methods=["POST"])
@app.route('/auth')
```

```javascript
router.post('/login', loginHandler);
app.post('/auth', authHandler);
```

```go
http.HandleFunc("/login", loginHandler)
r.POST("/login", handleLogin)
```

...when none of the following rate-limiting indicators are present anywhere in the same file: `@ratelimit`, `rateLimit(`, `rateLimiter`, `throttle(`, `limiter.Allow`, `redis.Incr`, `account_locked`, `lockout`, `max_attempts`, `maxAttempts`.

## Remediation

1. **Add rate limiting middleware** at the application or API gateway level.

```python
# SAFE: Flask-Limiter decorating the login route
from flask_limiter import Limiter
limiter = Limiter(app, key_func=get_remote_address)

@app.route("/login", methods=["POST"])
@limiter.limit("10 per minute")
def login():
    ...
```

```javascript
// SAFE: express-rate-limit on login endpoint
const rateLimit = require('express-rate-limit');
const loginLimiter = rateLimit({
    windowMs: 15 * 60 * 1000, // 15 minutes
    max: 10,
    message: 'Too many login attempts, please try again later',
});
app.post('/login', loginLimiter, loginHandler);
```

2. **Implement account lockout** after N consecutive failures (typically 5–10), with an exponential back-off or a time-limited lock.

```python
# SAFE: track failed attempts in Redis
MAX_ATTEMPTS = 5
LOCKOUT_DURATION = 300  # seconds

def check_lockout(username):
    key = f"login_attempts:{username}"
    attempts = redis_client.get(key)
    if attempts and int(attempts) >= MAX_ATTEMPTS:
        raise AccountLockedError("Account temporarily locked")
```

3. **Consider multi-factor authentication (MFA).** Even without rate limiting, MFA drastically reduces the value of a guessed password.

4. **Use CAPTCHA or proof-of-work** for high-value login endpoints to raise the cost per attempt for automated tools.

5. **Log and alert on anomalous login patterns.** Even with rate limiting in place, monitoring for distributed credential-stuffing (many IPs, many usernames) enables early detection.
