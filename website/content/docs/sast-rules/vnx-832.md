---
title: "VNX-832 – Unlock of a Resource That is Not Locked"
description: "Detect mutex unlock/release calls without a corresponding preceding lock/acquire, which can cause double-unlock panics or incorrect synchronization."
---

## Overview

This rule flags unlock operations (`mutex.Unlock()` in Go, `lock.release()` in Python, `lock.unlock()` in Java) that are not preceded by a corresponding lock acquisition within the visible code window. Unlocking an already-unlocked mutex panics in Go, causes a `RuntimeError` in Python, and produces an `IllegalMonitorStateException` in Java. In some cases it can also over-unlock, allowing concurrent access that was supposed to be serialized. This maps to [CWE-832: Unlock of a Resource That is Not Locked](https://cwe.mitre.org/data/definitions/832.html).

**Severity:** Medium | **CWE:** [CWE-832](https://cwe.mitre.org/data/definitions/832.html)

## Why This Matters

Double-unlock crashes are denial of service vectors in production services. More subtly, if a code path unlocks a mutex that was never locked in that path, the intended critical section is unprotected — concurrent goroutines or threads can enter simultaneously, causing data corruption or race conditions.

## What Gets Flagged

```go
// FLAGGED: Unlock without preceding Lock in this code path
func cleanup(mu *sync.Mutex) {
    defer mu.Unlock()  // called even if Lock was never called
    // ... cleanup work
}
```

```python
# FLAGGED: release() without acquire() — or acquire in a different branch
import threading
lock = threading.Lock()

def process():
    if condition:
        lock.acquire()
        do_work()
    lock.release()  // release called even when acquire was not
```

```java
// FLAGGED: unlock without lock
ReentrantLock lock = new ReentrantLock();

void process() {
    try {
        doWork();
    } finally {
        lock.unlock();  // unlock without preceding lock()
    }
}
```

## Remediation

```go
// SAFE: lock and defer unlock together, always paired
func process(mu *sync.Mutex) {
    mu.Lock()
    defer mu.Unlock()  // always paired with Lock()
    // critical section
}
```

```python
# SAFE: use context manager to guarantee acquire/release pairing
import threading
lock = threading.Lock()

def process():
    with lock:  # acquire on entry, release on exit — always paired
        do_work()

# SAFE: or explicitly paired acquire/release
def process():
    lock.acquire()
    try:
        do_work()
    finally:
        lock.release()  # guaranteed to run, always after acquire
```

```java
// SAFE: lock/unlock always paired
ReentrantLock lock = new ReentrantLock();

void process() {
    lock.lock();
    try {
        doWork();
    } finally {
        lock.unlock();  // always after lock()
    }
}
```

Always pair lock acquisition and release. Use language-provided constructs that guarantee pairing: `with` statements in Python, `defer mu.Unlock()` immediately after `mu.Lock()` in Go, and `try/finally` blocks in Java.
