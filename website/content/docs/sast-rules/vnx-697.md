---
title: "VNX-697 – Incorrect Comparison (PHP Type Juggling)"
description: "Detect PHP loose equality (==) comparisons of security-sensitive variables that can be bypassed through type coercion."
---

## Overview

PHP's loose comparison operator `==` performs type coercion before comparing. This enables well-known authentication bypasses: strings beginning with `0e` are treated as scientific notation floats (so `'0e1234' == '0e5678'` is true), and non-numeric strings compare equal to `0`. Attackers can exploit this to bypass password hash comparisons, token validation, and access control checks.

**Severity:** High | **CWE:** [CWE-697 – Incorrect Comparison](https://cwe.mitre.org/data/definitions/697.html)

## Why This Matters

The canonical exploit is against MD5/SHA1 hash comparisons: if both the stored and provided hash begin with `0e`, PHP considers them numerically equal (both equal `0` as a float). This has been exploited against login systems using weak hash storage.

## What Gets Flagged

```php
<?php
// FLAGGED: Type-juggling bypass
if ($password == 0) { /* any non-numeric string matches */ }
if ($token == true) { /* any non-empty, non-zero value matches */ }
if ($_GET['hash'] == $stored_hash) { /* 0e bypass */  }
```

## Remediation

Always use strict equality for security comparisons:

```php
<?php
// SAFE: strict equality
if ($password === $expected) { ... }

// SAFE: constant-time comparison for hashes/tokens
if (hash_equals($stored_hash, $provided_hash)) { ... }
```
